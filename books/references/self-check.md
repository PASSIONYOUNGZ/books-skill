# 自检标准

## 单本自检

每本书标记完成前，必须检查：

1. **内容质量**
   - 是否覆盖原书主要章节或主要内容群？
   - 是否保留方法、案例、练习、论证和风险提醒？
   - 各章是否有真实差异，而不是模板填空？
   - 读者不看原书，仅看整理稿，是否能学到原书的核心知识、观点、思想和方法？
   - 每个主要章节是否讲清“问题、观点、方法、例子、限制”，而不是只有关键词和概括句？
   - 是否先完成了内容地图，识别了原书的核心观点、方法、案例、练习、重复内容和弱抽取区域？
   - 长章节是否去掉了重复和空话，短章节是否补足了必要的方法、案例、边界或源文件限制说明？
   - 方法、模型、脚本或示例表达是否讲清目标、场景、步骤、反馈信号、失败处理和风险边界？
   - 重要案例是否讲清情境、做法、结果和作者想证明的结论？
2. **忠实度**
   - 是否避免编造原书没有的观点？
   - 是否区分作者观点和整理者的必要提醒？
   - 是否没有把个别案例过度泛化？
3. **结构**
   - 是否有四个顶层结构？
   - 标题、正文、附录、Q&A、练习是否处理得当？
   - 是否排除了广告、重复 TOC、抽取痕迹？
4. **语言与排版**
   - 语言是否流畅可读？
   - 标题层级是否稳定？
   - 加粗是否能帮助阅读，而不是制造噪声？
   - 列表是否只用于步骤、练习、框架和检查？
   - 是否删掉了“可以帮助读者”“有助于我们理解”“本节围绕”等没有信息量的套话？
   - 是否避免同一观点或同一案例在多个章节整段重复？
   - 是否清除了目录点线、长串省略号、行首页码、连续下划线、连续横线、广告尾巴和明显 OCR 噪声？
   - 是否有足够的重点加粗，帮助读者快速抓住核心概念、方法、模型和风险提醒？
   - 脚本库、方法书、教材类内容是否用稳定的小标题区分观点、步骤、案例、边界，而不是整段堆在一起？
5. **EPUB**
   - EPUB 是否存在？
   - 是否包含 `mimetype`、`EPUB/nav.xhtml`、`EPUB/chapter.xhtml`、`EPUB/content.opf`？
   - 是否保留标题、列表、加粗？

## 不合格与重做

出现以下任何问题，标记为不合格并重写或扩写：

- 相对原书体量明显过薄。
- 遗漏主要章节。
- 语言空泛、重复、模板化。
- 看起来很长，但主要由重复段落、原文堆叠、目录片段或泛泛提醒构成。
- 过短，且没有说明是原书材料少、抽取受限，还是确实已经覆盖关键内容。
- 章节整理只有关键词、主题归类或固定句式，没有原书的具体思想和方法。
- 案例、练习、脚本、清单在原书中很重要，但整理稿没有说明如何使用、适用边界和失败信号。
- 方法或脚本只有原句，没有说明场景、意图、机制、反馈和边界。
- 案例只说“举了一个案例”，没有拆出情境、做法、结果和结论。
- 只做“关于本书讲某个主题”的外部评论，没有真正进入原书内容。
- 正文出现工具痕迹或抽取日志。
- Markdown 层级混乱。
- Markdown 正文保留 `………………………………………………………………………`、大量 `......`、`__________`、`- 25 -` 或类似视觉噪声。
- 几乎没有关键词加粗、方法加粗、模型加粗或关键提醒，导致阅读时没有重点。
- EPUB 缺失或结构错误。
- 扫描版或劣质 OCR 被硬凑成整理。

必须修复后再进入下一本。

## 批量最终检查

收尾时用 manifest 对照输出：

- 每个源文件只有一个状态。
- 已完成书籍必须有 Markdown、EPUB、自检。
- 扫描版放弃书籍必须有放弃说明。
- 排除书籍必须写入总报告。
- 输出标题不能重复。
- 运行 `scripts/content_quality_audit.py`；对重复率高、正文过短、模板短语多、方法/案例/边界词密度低的文件做代理自审和自动返工。

## 自动处理规则

- 有红旗：默认不交付，先返工。返工方式包括回读原文、扩写方法和案例、删除重复、重组章节、清除工具痕迹。
- 有警告：默认由代理判断。若警告反映真实缺口，修；若源书本身短、案例少或抽取受限，在自检中说明，不要求用户裁决。
- 无红旗无警告：仍抽样复查内容密度和忠实度，避免脚本漏判。
- 质量与数量冲突：默认牺牲数量保质量，不询问用户；在批量报告中说明取舍即可。
- 长短取舍、重复删减、章节重组、重点书优先级、是否重写：默认由代理判断并执行。
- 源文件缺失、权限不足、需要联网或安装依赖、不可逆操作、互斥目标、敏感边界不清：默认不打断流程。能降级处理就降级，不能处理就记录阻断或放弃原因，继续其它项目，最后统一报告。
- 平台或运行环境强制要求批准时，优先选择不需要批准的替代路径；若没有替代路径，则记录为受限未处理并继续，而不是中断整批工作。
