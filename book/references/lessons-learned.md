# 通用使用问题与优化记录

## 抽取与清单完整性

### 常见问题

1. 批量处理时，输出目录、缓存目录或导出目录可能被误当成源书再次扫描，造成重复处理。
2. EPUB、HTML 或压缩包类格式可能因为目录名、文件名或内部路径匹配规则过宽而被漏收录。
3. PDF 可能是扫描版、图片版、混合版或文字层损坏版；只看文件扩展名不能判断是否可总结。
4. 一本书可能同时存在多个格式或重复副本，需要在 manifest 中稳定记录，避免重复输出。

### 通用规则

- 抽取后必须用 `manifest.json` 或等价清单核对源文件数量、状态和输出文件。
- 跳过规则应优先匹配目录角色，而不是简单匹配文件名片段。
- 扫描版或低质量 OCR 不能硬凑摘要；应写明放弃原因或等待用户提供更可靠来源。
- 每个源文件只应有一个最终状态：completed、abandoned、excluded、unsupported 或 failed。

## 内容质量

### 常见问题

1. 自动整理容易变成“标题扩写”：只看目录或章节标题，生成看似完整但没有原书细节的摘要。
2. 有些整理稿篇幅很长，但长在重复段落、原文堆叠、目录残留和泛泛提醒上。
3. 有些整理稿过短，只给结论或主题归纳，没有讲清原书的方法、案例、证据、练习、故事或判断标准。
4. 固定小标题会制造“看起来完整”的错觉：每章都有相同模板，但内容没有真实差异。
5. 脚本库、访谈集、论文集、案例集、小说和教材需要不同整理方式，不能用同一套章节模板机械处理。

### 通用规则

- 写正文前先建立内容地图：结构、主张、方法、案例、证据、练习、重复内容、弱抽取区域。
- 每段都应承担明确功能：解释观点、说明方法、拆解案例、复述关键情节、提示边界或连接章节递进。
- 章节长短按原书信息密度分配；重点章节、方法密集章节、案例密集章节应明显更详细。
- 脚本或表达模板要讲清场景、意图、机制、反馈信号、失败处理和边界，不要只堆原句。
- 小说或叙事类作品应保留人物、冲突、情节推进、主题和叙事手法，不要改写成管理学式要点。

## 排版与导出

### 常见问题

1. PDF/EPUB 抽取会把目录页点线、页码、广告尾巴、重复版权页和网页导航保留下来。
2. Markdown 只有标题和普通正文时，即使内容较长，也难以扫读出核心框架、关键概念和风险提醒。
3. 结构审计通过不代表内容合格；文件存在、EPUB 可打开，只说明导出流程成功。

### 通用规则

- EPUB 导出前搜索并清理 `…{6,}`、`\.{6,}`、`_{6,}`、`-{6,}`、`—{6,}` 等长标点或视觉噪声。
- 核心概念、模型名、方法名、关键主题和重要提醒应适度加粗；不要整段加粗。
- 先清理 Markdown，再生成 EPUB，避免脏排版进入电子书。
- 审计脚本只能当红旗系统；最终合格标准是读者能否不打开原书也学到原书的主要内容和判断方式。

## 自治质量控制

### 常见问题

1. 如果把红旗和警告都交给用户判断，会把质量责任转移给用户。
2. 批量任务容易为了数量牺牲质量，导致每本书都完成了文件，但没有真正完成整理。
3. 依赖、权限、OCR、源文件损坏和格式不支持等问题可能中断整批工作。

### 通用规则

- 红旗默认阻断导出并由代理自动返工；警告默认由代理自审并决定修复或在自检中说明。
- 质量与数量冲突时，默认选择质量；可以减少本轮处理数量，但必须在报告中说明。
- 遇到单本硬阻断时，记录 blocked/abandoned 原因并继续其它可处理书籍。
- 常规质量取舍、扩写/压缩、去重、结构重组、排版清理和 EPUB 重建都由代理自行执行。
- 只有用户目标互斥、需要权限审批、缺少必须源文件、必须联网安装依赖且无替代路径时，才中断询问或请求批准。

## 跨环境适配

- 不假设固定操作系统、固定用户名、固定项目目录或固定 Python 路径。
- 优先使用当前环境已有的 Python、shell、文档工具和解析库。
- Windows 上遇到中文编码问题时，可使用 UTF-8 模式运行验证或脚本；POSIX 环境按本地 shell 习惯执行。
- 缺少可选依赖时，先降级处理可读格式；不能可靠抽取时，记录限制，不编造内容。
